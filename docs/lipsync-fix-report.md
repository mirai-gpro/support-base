# リップシンク対策レポート（バックエンド → フロントエンド共有）

**作成日:** 2026-03-04
**最終更新:** 2026-03-05（フロントエンドフィードバック反映）
**対象:** support-base（バックエンド）
**共有先:** フロントエンドチーム

## 概要

Live API 導入に伴い、リップシンク（A2E: Audio to Expression）に複数の課題が発生しました。
本ドキュメントでは、バックエンド側で実施済みの対策をまとめます。

2026-03-05: フロントエンドチームからのフィードバックを受けて、
提案 #7 を段階的チャンクサイズ方式に変更し、追加要望 3件（P1〜P3）を実装しました。

---

## 背景：REST API → Live API で何が変わったか

### REST API（旧・gourmet-support）

```
ユーザー発話 → AI応答生成 → TTS一括変換 → A2E呼び出し（1回）
                                              ↓
                                         数秒の間隔
                                              ↓
ユーザー発話 → AI応答生成 → TTS一括変換 → A2E呼び出し（1回）
```

- 1ターンにつき A2E 呼び出し **1回**
- リクエスト間隔は数秒〜数十秒
- 503エラーは発生しなかった

### Live API（現在・support-base）

```
Gemini音声ストリーム開始
  → 0.25秒後  A2E呼び出し #1（初回チャンク）
  → 2.0秒後   A2E呼び出し #2
  → 5.0秒後   A2E呼び出し #3
  → ...（5秒ごと）
  → turn_complete → A2E呼び出し #final（残りバッファ）
```

- 段階的チャンクサイズにより呼び出し回数を大幅削減
- 10秒の応答で **4回**（旧: 11回）

## 各対策の詳細

### 対策 #1: A2E 503リトライ

**ファイル:** `support_base/services/a2e_client.py`

- 503 レスポンス受信時のみ、最大3回リトライ
- リトライ間隔: 0.3秒 → 0.6秒（指数バックオフ）
- 503以外のエラー・タイムアウトは即座にリターン（リトライしない）
- 音声再生をブロックしない設計

### 対策 #2: ストリーミング中の逐次A2E送信

**ファイル:** `support_base/live/relay.py`

- 変更前: `turn_complete` まで音声を全バッファリング → A2E一括呼び出し
- 変更後: 閾値（チャンクサイズ）に達するたびに `asyncio.create_task()` で非同期送信
- 音声再生中に表情データが届くようになった

### 対策 #3: 初回チャンクの即発火 + 段階的チャンクサイズ (#7)

**ファイル:** `support_base/live/relay.py`

```python
A2E_FIRST_CHUNK  = 24000 * 2 // 4   #  12,000 bytes = 0.25秒
A2E_SECOND_CHUNK = 24000 * 2 * 2    #  96,000 bytes = 2.0秒
A2E_NORMAL_CHUNK = 24000 * 2 * 5    # 240,000 bytes = 5.0秒
```

- chunk#1（0.25秒）: 口の動き開始を最速化
- chunk#2（2.0秒）: 冒頭2.25秒で2チャンク分が届き品質向上
- chunk#3〜（5.0秒）: 503対策として呼び出し頻度を削減
- フロントエンド逆提案を採用した段階的方式

### 対策 #4: 末尾フラッシュ＋パディング

**ファイル:** `support_base/live/relay.py`

- `turn_complete` 受信時にバッファ内の残り音声を全て送信
- 0.25秒未満の短いチャンクは無音（`\x00`）でパディング
- A2Eが処理可能な最低サイズを保証

### 対策 #5: is_start / is_final フラグ + sample_rate

**ファイル:** `support_base/live/relay.py`, `support_base/services/a2e_client.py`

- A2Eリクエストに以下を追加:
  - `is_start`: ターン内の最初のチャンクかどうか
  - `is_final`: ターン内の最後のチャンクかどうか
  - `sample_rate=24000`: PCM音声のサンプルレート明示
- これにより全フレームがゼロ値になる問題を解消

### 対策 #6: REST TTS の LINEAR16 変換

**ファイル:** `support_base/rest/router.py`

- TTS生成時に MP3（再生用）とは別に LINEAR16 PCM（A2E用）を生成
- A2Eへの送信時に `audio_format="pcm"`, `sample_rate=24000` を明示
- フォーマット不一致による表情生成失敗を解消

### 対策 #8: turn_complete 後のゼロチャンク送信停止（P1）

**ファイル:** `support_base/live/relay.py`

- `turn_complete` 時に `a2e_turn_complete = True` を設定
- 以降の非同期 A2E タスクが完了しても expression を送信しない
- 最終チャンク（`is_final=True`）のみ例外的に送信を許可
- これにより偽ストリーム開始と Turn2 データとの同期ずれを防止

### 対策 #9: expression に chunk_index / is_final を追加（P2）

**ファイル:** `support_base/live/relay.py`

- expression メッセージに `chunk_index`（0始まり連番）と `is_final`（最終チャンクフラグ）を追加
- フロントエンドでの正確なストリーム終了判定が可能に（「2秒間新着なし」のタイムアウト不要）

### 対策 #10: jawOpen スケーリング（P3）

**ファイル:** `support_base/live/relay.py`, `support_base/rest/router.py`

- A2E結果の `jawOpen` 値を **1.8倍** にスケーリング（上限1.0でクランプ）
- 平均値を 0.03 → 0.054〜0.06 に引き上げ（目標 0.08〜0.12）
- Live API / REST 両経路で適用
- スケール係数は `JAW_OPEN_SCALE = 1.8` で一元管理

## A2E呼び出し回数の比較（段階的チャンクサイズ適用後）

| AI応答の長さ | 旧（1秒均一） | 現在（段階的） |
|-------------|-------------|-------------|
| 10秒 | 11回 | 4回 |
| 15秒 | 16回 | 4回 |
| 20秒 | 21回 | 5回 |

### タイムライン

```
旧（1秒チャンク）:
  [0.25s] [1s] [1s] [1s] [1s] [1s] [1s] [1s] [1s] [1s] [残り]
  → 11回（10秒の応答時）

現在（段階的チャンク）:
  [0.25s] [2.0s] [5.0s] [2.75s残り]
  → 4回（10秒の応答時）
```

### 変更点まとめ

| 項目 | 変更前 | 変更後 |
|------|--------|--------|
| チャンクサイズ | 均一1秒 | 段階的（0.25s→2.0s→5.0s） |
| `chunk_index` | なし | **追加**（0始まり連番） |
| `is_final` | なし | **追加**（最終チャンクフラグ） |
| turn_complete 後の expression | 送信される（バグ） | **送信停止** |
| jawOpen の値 | 平均0.03 | **1.8倍スケーリング** |
