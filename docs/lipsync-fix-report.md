# リップシンク対策レポート（バックエンド → フロントエンド共有）

**作成日:** 2026-03-04
**対象:** support-base（バックエンド）
**共有先:** フロントエンドチーム

---

## 概要

Live API 導入に伴い、リップシンク（A2E: Audio to Expression）に複数の課題が発生しました。
本ドキュメントでは、バックエンド側で実施済み・提案中の対策をまとめます。

フロントエンドチームにて確認・フィードバックをお願いします。

---

## 背景：REST API → Live API で何が変わったか

### REST API（旧・gourmet-support）

```
ユーザー発話 → AI応答生成 → TTS一括変換 → A2E呼び出し（1回）
                                              ↓
                                         数秒の間隔
                                              ↓
ユーザー発話 → AI応答生成 → TTS一括変換 → A2E呼び出し（1回）
```

- 1ターンにつき A2E 呼び出し **1回**
- リクエスト間隔は数秒〜数十秒
- 503エラーは発生しなかった

### Live API（現在・support-base）

```
Gemini音声ストリーム開始
  → 0.25秒後  A2E呼び出し #1（初回チャンク）
  → 1.0秒後   A2E呼び出し #2
  → 1.0秒後   A2E呼び出し #3
  → ...（音声が続く限り毎秒）
  → turn_complete → A2E呼び出し #final（残りバッファ）
```

- 1ターンにつき A2E 呼び出し **10〜20回**
- リクエスト間隔は **1秒**（バースト的）
- Cloud Run の同時接続上限に到達 → **503エラー発生**

## 対策一覧

| # | 課題 | 対策 | 状態 |
|---|------|------|------|
| 1 | A2E 503エラー（Cloud Run過負荷） | A2Eリクエストに503専用リトライ追加（最大3回、0.3s/0.6s間隔） | **実装済** |
| 2 | 表情データが音声再生後にまとめて届く | `turn_complete` 待ちをやめ、音声ストリーム中に逐次A2E送信 | **実装済** |
| 3 | 初回の口の動き開始が遅い（〜500ms） | 最初のチャンクを0.25秒（12KB）で即発火し、初動を最速化 | **実装済** |
| 4 | ターン末尾の表情が欠落する | `turn_complete` 時にバッファ残量をフラッシュ＋無音パディング（最低0.25秒） | **実装済** |
| 5 | 表情が全てゼロ値になる | A2Eリクエストに `is_start`/`is_final` フラグと `sample_rate=24000` を追加 | **実装済** |
| 6 | REST TTS側の表情生成失敗 | TTS音声をLINEAR16(PCM)に変換してからA2Eへ送信 + sample_rate明示 | **実装済** |
| 7 | **A2E呼び出し頻度が高すぎる（503の根本原因）** | **通常チャンクを1秒→5秒に変更** | **未実装（提案）** |

## 各対策の詳細

### 対策 #1: A2E 503リトライ

**ファイル:** `support_base/services/a2e_client.py`

- 503 レスポンス受信時のみ、最大3回リトライ
- リトライ間隔: 0.3秒 → 0.6秒（指数バックオフ）
- 503以外のエラー・タイムアウトは即座にリターン（リトライしない）
- 音声再生をブロックしない設計

### 対策 #2: ストリーミング中の逐次A2E送信

**ファイル:** `support_base/live/relay.py`

- 変更前: `turn_complete` まで音声を全バッファリング → A2E一括呼び出し
- 変更後: 閾値（チャンクサイズ）に達するたびに `asyncio.create_task()` で非同期送信
- 音声再生中に表情データが届くようになった

### 対策 #3: 初回チャンクの即発火

**ファイル:** `support_base/live/relay.py`

```python
A2E_FIRST_CHUNK = 24000 * 2 // 4   # 12,000 bytes = 0.25秒
A2E_NORMAL_CHUNK = 24000 * 2        # 48,000 bytes = 1.0秒
```

- 最初のチャンクは 0.25秒分（12KB）で即座にA2Eへ送信
- 2回目以降は通常サイズ（現在1.0秒）で送信
- 口の動き開始までの遅延を最小化

### 対策 #4: 末尾フラッシュ＋パディング

**ファイル:** `support_base/live/relay.py`

- `turn_complete` 受信時にバッファ内の残り音声を全て送信
- 0.25秒未満の短いチャンクは無音（`\x00`）でパディング
- A2Eが処理可能な最低サイズを保証

### 対策 #5: is_start / is_final フラグ + sample_rate

**ファイル:** `support_base/live/relay.py`, `support_base/services/a2e_client.py`

- A2Eリクエストに以下を追加:
  - `is_start`: ターン内の最初のチャンクかどうか
  - `is_final`: ターン内の最後のチャンクかどうか
  - `sample_rate=24000`: PCM音声のサンプルレート明示
- これにより全フレームがゼロ値になる問題を解消

### 対策 #6: REST TTS の LINEAR16 変換

**ファイル:** `support_base/rest/router.py`

- TTS生成時に MP3（再生用）とは別に LINEAR16 PCM（A2E用）を生成
- A2Eへの送信時に `audio_format="pcm"`, `sample_rate=24000` を明示
- フォーマット不一致による表情生成失敗を解消

## 提案 #7: チャンク間隔を 1秒 → 5秒 に変更

### 目的

503エラーの**根本原因**（A2E呼び出し頻度の高さ）を解消する。

### 変更内容

```python
# 変更前
A2E_NORMAL_CHUNK = 24000 * 2        # 48,000 bytes = 1.0秒

# 変更後
A2E_NORMAL_CHUNK = 24000 * 2 * 5    # 240,000 bytes = 5.0秒
```

### A2E呼び出し回数の比較

| AI応答の長さ | 1秒（現在） | 5秒（変更後） |
|-------------|-----------|-------------|
| 10秒 | 11回 | 3回 |
| 15秒 | 16回 | 4回 |
| 20秒 | 21回 | 5回 |

### タイムライン比較

```
変更前（1秒チャンク）:
  [0.25s] [1s] [1s] [1s] [1s] [1s] [1s] [1s] [1s] [1s] [残り]
  → 11回の A2E 呼び出し（10秒の応答時）

変更後（5秒チャンク）:
  [0.25s] [5s] [4.75s残り]
  → 3回の A2E 呼び出し（10秒の応答時）
```

### 既存対策との関係

| 懸念事項 | 影響 |
|---------|------|
| 初回遅延 | **影響なし** — 初回チャンク（0.25秒即発火）は変更なし |
| 末尾欠落 | **影響なし** — `turn_complete` フラッシュは変更なし |
| 503リトライ | **併用可能** — 頻度低減 + リトライで二重の対策 |

## フロントエンドへの影響

### WebSocket メッセージフォーマット（変更なし）

```json
{
  "type": "expression",
  "data": {
    "names": ["browInnerUp", "browDownLeft", "...（52ブレンドシェイプ）"],
    "frames": [[0.12, 0.0, "..."], ["..."]],
    "frame_rate": 30
  }
}
```

### 変更される点

| 項目 | 変更前（1秒チャンク） | 変更後（5秒チャンク） |
|------|---------------------|---------------------|
| `expression` メッセージの送信頻度 | 約1秒ごと | 約5秒ごと |
| 1メッセージあたりのフレーム数 | 約30フレーム（1秒分） | 約150フレーム（5秒分） |
| メッセージフォーマット | — | **変更なし** |
| 初回メッセージの到着タイミング | — | **変更なし（0.25秒後）** |

### フロントエンド側の変更は不要と想定

`queueLiveExpressionFrames()` で受信フレームを順次キューイングし再生する設計であれば、
1回あたりのフレーム数が増えても動作に影響しないと想定しています。

## フロントエンドチームへの確認事項

以下の点についてご確認をお願いします：

1. **バッファサイズ**: `queueLiveExpressionFrames()` で150フレーム（5秒分）を一括受信しても問題ないか？
2. **再生タイミング**: 表情フレームの再生開始は受信直後から順次再生する想定で合っているか？（5秒分溜まってから一括再生、ではなく）
3. **中断時の挙動**: `interrupted` メッセージ受信時、キュー内の未再生フレームはクリアする実装になっているか？
